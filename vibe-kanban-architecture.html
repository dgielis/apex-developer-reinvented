Every Lane Transition & Its Side Effects

Comprehensive table of what happens when a ticket is dragged from one lane to another.

From To Trigger Automated Side Effects
— (new) Backlog Issue created Background LLM enrichment spawns asynchronously. Classifies the issue into category (Bug/Feature/Task), type (packages/apex_app/db_objects), time_estimate (30m to 1w), and suggests an assignee. Results stored in extension_metadata. Non-blocking — UI remains responsive.
Backlog To Do Drag & drop Planning agent spawns. An ExecutionProcess is created with a planning prompt. The AI agent: (1) reads the issue title and description, (2) explores the codebase and database schema, (3) generates a PLAN.md file. Sets planning_status: "running" and stores planning_workspace_id + planning_execution_id. Backend polls execution status. On success: extracts PLAN.md, stores in metadata, auto-moves to "Planning Done". On failure: sets planning_status: "failed", issue stays.
To Do Planning Done Automatic (on plan success) Automatic transition. planning_status set to "done". The plan markdown is readable/editable in the UI. No additional processes spawn. This is a human checkpoint — the user reviews the plan before committing to coding.
Planning Done In Progress Drag & drop Coding agent spawns with full Oracle PDB lifecycle. Requires a plan in extension_metadata.plan (silently fails without). SetupScript phase: (1) Creates Docker container (local-{repo}-dev), (2) Clones Oracle PDB from freepdb1 template: CREATE PLUGGABLE DATABASE DEV_{id}_{rand} FROM freepdb1, (3) Opens PDB read-write and saves state, (4) Resets wk_app_app user credentials, (5) Maps Docker port 1521 to host, (6) Creates SQLcl named connection ({repo}-dev) pointing to the new PDB, (7) Configures Claude MCP override for SQLcl access, (8) Updates .gitignore. CodingAgent phase: Agent connects via named connection and implements the plan against the isolated PDB. Validation phase: Re-exports from PDB (project export) and checks git status for drift. Cleanup phase: ALTER PLUGGABLE DATABASE ... CLOSE; DROP PLUGGABLE DATABASE ... INCLUDING DATAFILES;, restores original named connection, removes MCP override. Sets coding_status: "running", stores pdb_name, coding_branch, coding_workspace_id. On success: auto-moves to "In Review". On failure: PDB still dropped, sets coding_status: "failed".
In Progress In Review Automatic (on coding success) Automatic transition. coding_status set to "done". Code diffs become visible in the UI (inline diffs up to 2MB). The ApprovalService is active — any tool usage by follow-up agents requires user approval. The QueuedMessageService allows chaining additional agent prompts while an execution is still running.
In Review Merging Drag & drop Merge agent spawns. (1) Runs cleanup script (CleanupScript context), (2) reads coding_branch from metadata, (3) creates a Pull Request from coding branch to develop, (4) agent generates a PR description, (5) stores merging_pr_url and merge_commit_sha. If Jenkins CI is detected in the repo, triggers a build and stores jenkins_build_url. Sets merging_status: "running". On success: auto-moves to "Merged to Dev". On failure: sets merging_status: "failed" with merging_error.
Merging Merged to Dev Automatic (on merge success) Automatic transition. The PR Monitor Service (polling every 60s) detects the PR has been merged and auto-archives the workspace. The Remote Sync Service pushes the final workspace state to the cloud (files changed, lines added/removed). The workspace's Docker container and dev database can be cleaned up.
Merged to Dev Ready to Release Drag & drop No automation. Simple status change. Issues collect in this lane as a batch gate. The merge_commit_sha in each issue's metadata is critical — it will be used in the release-to-test cherry-pick operation.
Ready to Release On Test "Release to Test" button Batch release operation (not individual drag). Endpoint: POST /projects/{id}/release-to-test. (1) Fetches ALL issues in "Ready to Release" status, (2) extracts each issue's merge_commit_sha, (3) fetches origin, (4) checks out/creates the test branch, (5) pulls latest test, (6) cherry-picks ALL merge commits onto test, (7) pushes to origin/test, (8) returns to develop. If Jenkins config exists: triggers test build, polls for result. On Jenkins success: batch-moves all issues to "On Test". On failure: issues stay in "Ready to Release". Without Jenkins: moves directly to "On Test".
On Test Done Drag & drop No automation. Manual confirmation that testing passed. Sets completed_at timestamp. Issue lifecycle complete.
Any lane Cancelled Drag & drop Issue is abandoned. The "Cancelled" lane is hidden from the kanban board (visible in list view only). Any running execution processes for the issue are not automatically killed — the user should stop them manually via the workspace UI.

Backend Services & Executor Architecture

⚙ Executor System

The pluggable engine that runs AI coding agents. All executors implement the same interface and can be swapped per workspace.

  • Claude Code — Anthropic (npx-based, v2.1.62)
  • Codex — OpenAI's coding agent
  • Gemini CLI — Google's AI agent
  • Cursor Agent — with MCP server support
  • GitHub Copilot — GitHub's assistant
  • Amp — session forking support
  • QwenCode — Alibaba's agent
  • OpenCode / Droid / ACP — additional agents
Action Types: CodingAgentInitialRequest — First prompt CodingAgentFollowUpRequest — Continue session ScriptRequest — Shell scripts ReviewRequest — Code review

⚡ EventService (Real-time)

Hooks into SQLite write operations and broadcasts JSON patches via Server-Sent Events (SSE).

  • Monitors: workspaces, execution_processes, sessions, coding_agent_turns, kanban tables
  • Pre-update hooks detect deletions and push removal patches
  • Update hooks push full record patches on INSERT/UPDATE
  • Kanban tables get lightweight notification patches (just timestamps)
  • Uses RwLock + broadcast channels for fan-out

🔍 PR Monitor Service

Background task that polls Git host providers every 60 seconds.

  • Checks status of all open PRs in the database
  • Supports GitHub, GitLab, and Bitbucket
  • Updates merge status when PR state changes (Merged / Closed / Declined)
  • Auto-archives workspace when all PRs are merged
  • Syncs PR status to remote cloud server

🔒 Approval Service

Gates executor actions that need human confirmation.

  • Tool approvals — User must approve before agent uses certain tools
  • Question approvals — Agent asks user questions mid-execution
  • DashMap-based in-memory tracking (pending + completed)
  • Timeout watcher auto-expires stale approvals
  • Patches broadcast to frontend for real-time UI updates

🔁 Queued Message Service

Chains multiple agent runs without user intervention.

  • queue_message() — Queue follow-up while execution is running
  • take_queued() — Consume after execution completes
  • Enables continuous agent iteration without blocking
  • One queued message per session at a time

📦 Container Service

Manages Docker containers and process lifecycle for workspaces.

  • start_execution() — Spawn process, create session
  • stop_execution() — Graceful stop with cancellation tokens
  • Tracks child processes, message stores, DB stream handles
  • Auto-cleanup: debounced 2-minute timer for old workspaces
  • Image cleanup service removes orphaned Docker images

☁ Remote Sync Service

Pushes local workspace state to the cloud for team visibility.

  • Syncs workspace metadata (name, archived, files changed, lines added/removed)
  • Syncs issue status from local workspace merge
  • Syncs PR information (status, commit SHA)
  • Configured via VK_SHARED_API_BASE env var
  • Graceful degradation — disabled if not configured

📃 Log Streaming Pipeline

Real-time log processing from executor to UI.

  • Executor stdout/stderr → MsgStore captures raw chunks
  • Log processors normalize entries (PlainText, per-executor)
  • Typed entries: ErrorMessage, FileChange, TodoItem, ToolStatus, AskQuestion
  • JSON patch generation with entry index tracking
  • Background DB tasks persist logs to filesystem (JSONL format)

Oracle Pluggable Database (PDB) — Full Lifecycle

When a ticket is dragged to "In Progress", an isolated Oracle PDB is cloned from a template so the coding agent has its own database sandbox. This is the complete lifecycle.

⚙ Why Pluggable DBs?

Each coding task gets a fully isolated Oracle PDB clone so the agent can make schema changes, create objects, run DDL/DML freely without affecting the shared template database (freepdb1) or any other developer's work. After the agent finishes, the changes are validated by re-exporting and the PDB is dropped — the schema changes live only in the committed Git files.

📋 Naming Convention

PDB Name Format: DEV_{first_8_hex_of_issue_uuid}_{4_random_hex} Example: Issue UUID: a1b2c3d4-e5f6-7890-... PDB Name: DEV_A1B2C3D4_F5E6 The random suffix prevents collisions when retrying (old PDB may still be dropping while the new one is created).

Complete PDB Lifecycle Flow

Key Technical Details

Template Database: freepdb1

The source PDB that every coding task clones from. It contains the full application schema — all tables, views, packages, triggers, sequences, and seed data. It is never modified directly; it serves as a read-only golden image. When the CREATE PLUGGABLE DATABASE ... FROM freepdb1 runs, Oracle performs a full clone of all datafiles, creating a byte-identical but independent database that the agent can freely modify.

SQLcl Named Connections

Instead of passing raw JDBC strings, the system uses SQLcl's named connection feature. During setup, it saves a connection called {repo}-dev that points to the newly cloned PDB at the mapped Docker port. The coding agent only needs to run connect {repo}-dev to get a fully authenticated session. During cleanup, this named connection is restored to point back at the original freepdb1 database so manual developer workflows aren't broken.

Re-Export Validation

After the coding agent finishes, the system reconnects to the PDB and runs project export — this dumps all database objects back to the filesystem. It then checks git status: if the re-exported files differ from what the agent committed, it means the PDB state drifted from the file-based source of truth. Known drift (like ref_constraints/ files that always differ between PDB clones) is automatically cleaned. If unexpected drift remains, coding is marked as failed.

Cleanup Guarantees

The PDB is always dropped, regardless of success or failure. The cleanup script runs as a separate phase (not chained to the agent's action) to ensure it executes even if the agent crashes. It closes the PDB with CLOSE IMMEDIATE INSTANCES=ALL and drops it with INCLUDING DATAFILES to reclaim disk space. If cleanup itself fails, a warning is logged but the process continues. On retry, a new random suffix is generated for the PDB name so there's no collision with a still-dropping old PDB.

MCP Override for Agent

The setup script writes a local .mcp.json file into the workspace directory. This configures the Claude Code agent's SQLcl MCP server to use the PDB-specific named connection instead of the default. This means when the agent uses SQL tools through MCP, all queries automatically target the isolated PDB. During cleanup, this override is removed with claude mcp remove sqlcl --scope local.

Connection Credentials

User: wk_app_app Password: Welcome_1 Port: 1521 (Docker-mapped to dynamic host port) Service: {pdb_name} (e.g., DEV_A1B2C3D4_F5E6) Host: localhost Full JDBC: wk_app_app/Welcome_1@localhost:{HOST_PORT}/{pdb_name} Agent verifies connection with: SELECT sys_context('USERENV','CON_NAME') FROM dual; → Must return the PDB name, NOT 'FREEPDB1'
